# 任务说明

根据用户逐条输入的 Java 编码规范，生成可用于静态检查验证的独立 Java 测试项目。

这些测试项目的目标不是保证代码能够正常运行，而是用于验证代码仓新增的静态检查能力能否识别指定规范相关的问题，包括但不限于代码风格问题、潜在严重缺陷问题，例如数组越界。

每一条规范都必须生成一套独立的测试模块，模块之间互不干扰，尽可能不依赖网络、硬盘、数据库、外部服务或特殊硬件资源。

# 总体目标

针对每一条输入规范，完成以下两类产物的生成：

1. 规范拆解产物  
   将规范拆解为完备的测试因子，并保存为 JSON 文件。

2. 测试用例代码产物  
   根据测试因子生成覆盖这些因子的 Java 样例代码，并组织成独立的 Maven 模块结构，供静态检查系统扫描。

# 全局约束

1. 每条规范必须独立处理。
2. 每个测试用例必须相互独立，互不干扰。
3. 尽可能避免依赖网络、磁盘、大量外部资源、数据库、中间件或特殊硬件。
4. 优先生成最小化、可理解、可定位违规点的 Java 项目。
5. 生成代码的核心目标是触发静态检查，不要求代码必须能够正常运行。
6. 模块之间的组织和管理必须使用 Maven。
7. 步骤一中生成的 JSON 文件，必须保存在对应规范目录下的 `0` 文件夹中。
8. JSON 必须严格遵循 `template.json` 的结构与字段命名。
9. 每条规范对应的目录、测试用例目录及相关产物目录，统一放在 `src/main/java` 下。
10. 每条规范对应的一级模块必须作为整个项目根模块（根 `pom.xml`）的直接子模块。
11. JSON 产物必须通过 Python 脚本生成，不允许手工拼装或手工编辑为主流程。
12. 每个用例模块的违规代码统一尽量保留一个 `org.example.Main` 类，非必要不生成额外业务类。
13. 当前仓库已存在的规范一级模块示例：`src/main/java/avoid_double_equals_for_objects`、`src/main/java/ensure_no_useless_object_references`；新增规范按同级目录扩展。

# 命名规则

针对每条规范，统一基于规范名称生成两种命名形式：

- `spec_snake_name`：规范名对应的英文蛇形命名，用于文件夹名称、JSON 文件名及结构化存储标识。
- `spec_camel_name`：规范名对应的英文驼峰命名，用于模块名称。

例如：

- 规范名：禁止数组越界访问
- `spec_snake_name`: avoid_array_index_out_of_bounds
- `spec_camel_name`: avoidArrayIndexOutOfBounds

命名约束如下：

1. 文件夹名称统一使用 `spec_snake_name`，且路径位于 `src/main/java/<spec_snake_name>`。
2. 模块名称统一使用 `spec_camel_name`。
3. JSON 中测试因子的结构与字段命名必须遵循 `template.json`。

# 失败处理

如果某条规范在生成测试因子或测试用例时存在明显困难、语义不清、无法稳定落地，允许暂时跳过，不强行生成。

但是：

1. 不能直接忽略这条规范。
2. 必须将未完成的规范记录下来。
3. 必须保存未完成原因和当前处理状态，便于后续补做。

# 步骤一：规范拆解与测试因子产物生成

## 步骤目标

从终端读取一条规范文本，对其进行分析，拆解出完备的测试因子，并基于这些测试因子生成完整的基础用例描述，随后将结果保存为 JSON 文件。

## 要做的事情

1. 从终端一次读取一条规范。
2. 用户先给出该规范的测试因子数量与因子名称。
3. 基于用户给定因子，补全可落地的因子取值、边界场景和用例描述。
4. 使用 Python 脚本将所有测试因子及其对应的用例描述保存为 JSON 文件（结构严格对齐 `template.json`）。
5. 将生成的 JSON 文件（以及对应生成脚本）保存到该规范目录下的 `0` 文件夹中。

## 输出要求

1. JSON 结构必须遵循 `template.json`，字段名和层级保持一致。
2. 测试因子拆解必须能直接支撑步骤二的代码生成，避免后续再次解释语义。

## 推荐的数据结构

统一按如下结构理解：

`factor_name -> [case_description_1, case_description_2, case_description_3, ...]`

其中：

- `factor_name` 是测试因子名；
- `case_description_n` 是该测试因子下的候选用例描述。

# 步骤二：测试样例代码生成

## 步骤目标

根据步骤一中拆解出的测试因子和用例描述，生成用于静态检查的 Java 样例代码，并组织成相互独立的 Maven 模块结构。

## 要做的事情

1. 按照基用例覆盖的要求生成测试代码。
2. 每个测试因子至少要被覆盖一次。
3. 对每条规范在 `src/main/java` 下创建一个新的顶层文件夹。
4. 该文件夹名称使用该规范名的蛇形命名，即 `spec_snake_name`，路径为 `src/main/java/<spec_snake_name>`。
5. 对每条规范创建一个新的一级模块。
6. 该模块名称使用该规范名的驼峰命名，即 `spec_camel_name`。
7. 该一级模块必须在根 `pom.xml` 的 `<modules>` 中声明为子模块。
8. 该一级模块目录下必须提供聚合 `pom.xml`，并在其 `<modules>` 中声明所有外层序号目录下的用例模块（可参考 `avoid_double_equals_for_objects/pom.xml` 与 `ensure_no_useless_object_references/pom.xml`）。
9. 在规范文件夹下，按规则为每个用例创建外层序号目录，序号计算方式为 `a*100 + b`。
   - `a`：该样例涉及到的第一个测试因子在 JSON 汇总测试因子列表中的序号（从 1 开始）；
   - `b`：该样例在该测试因子下的序号（从 1 开始）。
10. 在每个外层序号目录下，再创建用例模块目录，模块目录名称使用“规范驼峰名 + 外层序号”的形式，例如：`avoidArrayIndexOutOfBounds101`。
11. 在每个内层用例模块目录内创建一个独立模块，不使用 `src` 作为用例顶层目录名。
12. 每个用例模块中只生成一个违规样例类：`org.example.Main`。
13. `Main.java` 必须放在内层用例模块目录下的 `src/main/java/org/example/Main.java`（即 `<spec_snake_name>/<outer_case_no>/<case_module_dir>/src/main/java/org/example/Main.java`）。
14. 每个样例都必须提供执行入口：`public static void main(String[] args)`。
15. 每个用例中的 Java 代码必须严格围绕该规范对应的测试因子来构造。
16. 代码不要求能够正常运行，但必须清晰表达出违反规范的具体方式。
17. **严格遵守 Javadoc 注释规范：**
    - 必须为公有类(public)、接口、类的公有和受保护属性(public/protected)、类的公有和受保护方法（含构造方法）添加 Javadoc 注释。
    - **豁免项：** 带有 `@Override` 的方法、使用 JUnit 框架默认的 `@BeforeClass` 和 `@AfterClass` 方法可以不加 Javadoc。
    - **顶层 public 类的 Javadoc：** 必须包含功能说明
    规范：直接引用输入的规范文本（例如：对象比较必须使用 equals 显式比较）。
    违规说明：描述代码具体是如何违规的（例如：在 if 条件中使用 “==” 比较对象引用）。
    测试意图：明确说明当前用例覆盖的具体因子分量（例如：覆盖比较对象类型：Short 包装类）。
    并按以下格式包含作者、版本和日期信息：
      ```java
      /**
       * 规范：该代码违反了哪条规范对象
       * 违规说明：违规点是什么
       * 测试意图：测试意图是什么
       *
       * @author 曹卓熠
       * @version 1.0.0
       * @since 2026-03-21 (使用当前生成日期)
       */
      ```
    - **方法的 Javadoc：** 应该包含功能说明，并按照实际需要，按顺序使用 `@param`、`@return`、`@throws` 标签对应说明。
    - 不需要添加注释的方法，无需添加空的、只有格式的注释骨架。
    - 注释符（如 `*` 或 `//`）与注释内容之间应有空格，注释与代码之间应视情况保留空行或空格。
18. **严格遵守代码日志输出规范：**
    - 禁止使用 `System.out.println` 等标准输出方式。
    - 必须使用 Lombok 配合 Slf4j 进行日志输出。在类上添加 `@Slf4j` 注解，代码中使用 `log.info(...)`、`log.error(...)` 等方式（如：`log.info("viol");`）。
19. **严格遵守检查点与测试因子内联注释规范：**
    - 测试因子的声明注释必须放置在最能体现该因子特征的代码行行尾，格式为：`// 测试因子(factor_key=value_key)`。
    - 违规点的声明注释必须放在直接触发规范违规代码（行尾），格式为：`// 检查点`。
    - **合并规则：** 如果仅当同一行代码既是因子体现点又是违规触发点时，这时注释必须合并在一行，格式为：`// 检查点, 测试因子(factor_key=value_key)`。
    - 一个用例只允许对应一条测试因子。
20. 所有模块及模块间组织必须通过 Maven 管理。


## 输出要求

1. 每个用例尽量只表达一个主要违规点，或一个最小必要的违规组合。
2. 非目标部分的代码尽量保持简洁、干净，减少无关噪音。
3. 代码的首要目标是让静态检查工具能够稳定识别出目标问题。
4. 不要为了让代码能运行而削弱违规特征。
5. 代码中违规点必须清晰、集中、可定位。
6. 插桩注释 `// 检查点` 必须紧跟在违规代码行的行尾，不可遗漏。
7. 每个规范对应的步骤一产物，也必须保存在该规范目录下的 `0` 文件夹中。
8. `// 测试因子(...)` 中的 `factor_key` 和 `value_key` 必须与该规范 JSON 文件中的测试因子键名及取值键名一致（以 `template.json` 的结构为准）。
9. `// 测试因子(...)` 必须真实反映该样例实际覆盖的测试因子，必须放置在对应代码行的行尾。不允许声明未覆盖的关键因子。
10. 外层目录编号必须能由 JSON 因子顺序与样例在对应因子下的序号反推得到，并与 `// 测试因子(...)` 保持一致。

# 生成策略要求

1. 优先保证静态检查可触发性，而不是运行正确性。
2. 优先保证违规点清晰可定位，而不是工程复杂度完整。
3. 优先保证最小必要样例，避免引入与目标规范无关的复杂依赖。
4. 除了故意违反目标规范的代码外，其余代码尽量保持整洁和最小化。
5. 对于难以构造的规范，不要伪造低质量样例，应记录为未完成项。

# 最终交付要求

对于每一条规范，最终至少应产出以下内容：

1. 对应规范的独立顶层文件夹，文件夹名使用规范名的蛇形命名，且位于 `src/main/java` 下；
2. 对应规范的独立模块，模块名使用规范名的驼峰命名，并在根 `pom.xml` 中声明为子模块；
3. 该规范拆解后的 Excel 文件（位于该规范目录下的 `0` 文件夹）；
4. 该规范拆解后的 JSON 文件（位于该规范目录下的 `0` 文件夹）；
5. 若干个按 `a*100+b` 命名的外层目录（如 `101`、`102`、`201`...）及其下按“规范驼峰名 + 外层序号”命名的内层测试用例模块目录；
6. 每个测试用例对应的独立 Maven 模块；
7. 每个用例模块内 `src/main/java/org/example/Main.java` 形式的 Java 违规样例代码，且模块构建后应包含 `target` 目录；
8. 未完成规范的记录清单（如有）。
9. 用于生成该规范 JSON 的 Python 脚本（或统一 Python 生成脚本及本规范配置数据，位于该规范目录下的 `0` 文件夹）。

# 禁止事项

1. 不要让不同规范的测试用例互相引用、互相依赖或互相污染。
2. 不要默认引入网络、数据库、磁盘文件、大型中间件等外部依赖。
3. 不要为了让代码通过编译或运行而主动修复目标违规点。
4. 不要漏掉 `// 检查点` 插桩。
5. 不要跳过无法完成的规范而不做记录。
6. 不要在 JSON 中使用与 `template.json` 不一致的测试因子结构或字段命名。
7. 不要生成与规范无关的大量样板代码。
9. 不要生成缺少 `// 测试因子(...)` 注释的样例代码。
10. 不要在 `// 测试因子(...)` 中使用与 JSON 不一致的因子名或取值名。
11. 严禁出现独立的、不在行尾的测试因子或检查点注释。必须按照行尾或行尾合并规则书写。
12. 不要在单行 `// 测试因子(...)` 中声明多个测试因子。
13. 不要将测试因子产物（JSON、生成脚本）存放在该规范目录的 `0` 文件夹之外。
14. 不要使用与 `a*100+b` 规则不一致的外层序号，或出现无法由 JSON 因子顺序与样例序号反推的编号。
15. 不要将 `Main.java` 放在 `src/main/java` 之外的目录。
16. 不要交付缺少 `target` 构建产物目录的用例模块。
17. 严禁在生成的代码中使用 `System.out.println`、`System.err.println` 等标准输出方式打印日志，必须使用 `@Slf4j`。
